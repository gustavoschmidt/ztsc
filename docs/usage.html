usage

Using ztsc

One binary, one command. ztsc is a batch checker: it reads your tsconfig.json (or explicit entry files), checks everything, prints tsc-style diagnostics, and exits.

01 · install

Build from source

v0.0.1 is not on npm yet. Once it is, bunx ztsc / npx ztsc will work with no install step. Until then, build from source — the only requirement is Zig 0.16.0. There are no dependencies to fetch: build.zig.zon lists none, and the source uses nothing but the Zig standard library.

git clone https://github.com/gustavoschmidt/ztsc
cd ztsc
zig build -Doptimize=ReleaseFast   # -> zig-out/bin/ztsc

A debug build is just zig build; zig build test runs the unit tests and the 414-case differential conformance suite.

02 · invocation

Point it at a project

ztsc                       # finds tsconfig.json in cwd or a parent
ztsc -p path/to/project    # explicit tsconfig (file or directory)
ztsc src/main.ts           # or explicit entry files

With no files, ztsc looks for tsconfig.json in the current directory and its parents. Imports and triple-slash references are discovered from the entry points automatically, including through node_modules packages and @types.

Options

flagwhat it does
-p, --project <path>use the tsconfig.json at <path> (file or directory)
--pretty[=true|false]tsc-style colored diagnostics with source excerpts (default: on when stderr is a TTY)
--verboseprint notes about accepted-but-ignored tsconfig options
--noLibskip the built-in esnext lib — no globals, no primitive/array methods (matches tsc)
--lib=a,b,cselect built-in lib blobs (es*, dom* families); overrides the tsconfig lib field
--skip-default-lib-checkdo not type-check the embedded lib files (checked by default, matching tsc/tsgo; tsc's skipDefaultLibCheck)
--workers=Nworker threads for the front end (default: CPU count)
--checkers=Nindependent checker instances (default: min(4, CPUs)); output is byte-identical for any N
-h, --helpprint help and exit
--versionprint version and exit

Introspection & benchmark aids

These exist because the project's performance claims are measured, not guessed — the same flags that produced the published numbers ship in the binary:

flagwhat it does
--timingper-phase wall-clock timings (load / scan / parse / bind / resolve / link / check)
--memoryarena and memory statistics: bytes per token / node / type, cache hit rates, heap total
--censustally out-of-subset constructs by frequency — run it on your codebase to see exactly what ztsc can't check yet
--dump-astS-expression parse trees (golden-test format)
--dump-symbolsbinder scope/symbol dumps
--dump-typesper-declaration checked types
--repeat=Nscan/parse/bind each file N times (benchmark aid)
--no-resolve-cachedisable the module-resolution memo (benchmark aid)
--no-frozen-storedisable the shared frozen base type store (benchmark aid / oracle)
--no-inst-cachedisable instantiation caching, re-run every substitution (benchmark aid / oracle)

03 · configuration

The tsconfig subset

ztsc reads the options that determine what to check; everything else is accepted and ignored (--verbose tells you exactly which options were ignored):

  • files, include / exclude — glob-driven file selection, tsc-compatible semantics.
  • baseUrl and paths — path-mapped module resolution.
  • Strict mode only. ztsc always checks with strictNullChecks-style semantics; there is no loose mode.

The standard library is built into the binary: the real TypeScript 7.0.2 lib.esnext reference chain (ES core through esnext) plus the real DOM lib (dom + dom.iterable + dom.asynciterable). The tsconfig lib field selects blobs (es* and dom* families; the default is ES-core + DOM, matching tsgo's target-esnext default); --lib=… overrides it and --noLib turns the lib off entirely.

By default the built-in standard library is type-checked, matching tsc and tsgo, which check their default lib at their defaults. The skipLibCheck and skipDefaultLibCheck tsconfig keys (accepted with a --verbose note rather than an unknown-option warning) — and the --skip-default-lib-check flag — turn that off: the embedded lib is then not type-checked, while your dependencies' .d.ts are still checked (so skipLibCheck is honored only partially). The CLI flag overrides the tsconfig keys. Diagnostics are byte-identical either way — lib diagnostics are never surfaced — so skipping only saves a few milliseconds and a few MB.

04 · output

Diagnostics and exit codes

Diagnostics match the TypeScript compiler's codes and messages, with tsc-style source excerpts when --pretty is on:

demo.ts:1:19 - error TS2322: Type 'string' is not assignable to type 'number'.

1 const n: number = "hi";
                    ~~~~

With --pretty=false you get one machine-friendly line per diagnostic. Either way, output is deterministic: diagnostics print per file in discovery order, position-sorted, and are byte-identical for any --workers / --checkers value — the conformance suite asserts it.

exit codemeaning
0no errors
1type or syntax errors reported
2usage, config, or file-system errors

05 · limitations

What it doesn't check yet

ztsc checks a large, well-defined subset of strict-mode TypeScript: the full type-level language (conditional types with infer, mapped types, template-literal types, generics, narrowing, declaration merging), the real ES-core…esnext + DOM standard library with the full iteration protocol (for…of over every DOM collection, for await with the sync-iterable fallback), CommonJS interop, const-symbol computed keys, and JSX against the real @types/react. What it checks, it checks like the TypeScript compiler. Every gap below fails in the safe direction: ztsc may miss an error tsc would report, but it never reports an error on valid code.

What it does not check yet:

  • Watch mode and LSP — ztsc is batch-only; both are planned next, on an architecture built for them.
  • tsconfig options beyond the subset — only files / include / exclude / baseUrl / paths / lib (plus the skipLibCheck keys) are honored, strict mode only (strict: false is refused); everything else is accepted and ignored, and --verbose lists which.
  • lib families other than es* and dom* (e.g. webworker) — they warn and are ignored.
  • Generator corners: yield* delegation is unchecked, and unannotated generator functions type as any.
  • CommonJS corners: a namespace import of an export = module keeps the export's call signature (ns() is not flagged), and a member of a require-bound namespace used in type position resolves to any.
  • Symbol-key corners: a plain non-unique symbol key (rxjs's [Symbol.observable]) is keyed by name rather than as a symbol index, and a deeper-qualified key ([a.b.c]) is out of subset.
  • JSX corners: prop type mismatches arriving inside a spread object, spreads of unions/generics/index-signature types, and children value typing are unchecked; class-component prop mistakes report refined codes where tsgo reports TS2769.
  • A handful of other known edge cases miss an error the TypeScript compiler would report.

Unsupported syntax produces a clear "not yet supported" diagnostic — never a wrong answer or a crash. Run ztsc --census on your project to get a frequency table of exactly which unsupported constructs it contains.
