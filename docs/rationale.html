rationale

Why ztsc exists

Because AI writes code fast, the tooling that checks that code has to be fast and cheap to run — that observation started this project. TypeScript 7 already rewrote the compiler in a native language and made it fast. ztsc makes a different bet: memory is where a systems language beats a garbage-collected one — and if you design for memory with total consistency, speed comes along for free.

01 · why now

Machines call the checker now

ztsc started from one observation: AI writes code fast, so the tooling that checks that code has to be fast and cheap to run. Stated that loosely it invites pushback, so here is the precise version. An agent typechecks after nearly every edit — dozens to hundreds of invocations per session, not a handful per day. And agents multiply: parallel worktrees, CI matrices, fleets of cloud sandboxes, all running checker instances simultaneously on the same hardware. The type checker stops being a tool a person runs occasionally and becomes infrastructure that machines call at high frequency and high concurrency. That shift changes which properties matter:

  • Memory per instance is a density limit. One 278 MB checker is nothing on a laptop. Fifty concurrent memory-capped sandboxes is where 278 MB vs 22 MB decides how many agents fit on a host — and what the GB-hours cost.
  • The per-invocation floor gets multiplied by thousands. A checker run once a day can shrug off startup cost; a checker run on every agent edit pays it every time. ztsc's entire process floor — start, load the embedded lib, check — is ~13 ms.
  • Determinism is a cache key. Byte-identical output at any parallelism means a harness can cache, diff, and dedupe checker results reliably. The batch CLI is the agent interface — which is why batch shipped before the LSP.
  • False positives are worse for agents than for people. A person shrugs off a bogus error; an agent dutifully "fixes" it and damages correct code. ztsc's contract — in its known gaps it under-reports, and it never errors on valid code — is exactly the contract a machine-called guardrail needs.

The honest caveat, stated before anyone else states it: inside a single interactive agent loop, checker latency is rarely the bottleneck — model inference is. The economics live in frequency × concurrency × cost: verification that is nearly free gets run more often, and instances that are memory-lean pack more densely. Speed is the supporting act; memory is the headline — which is the next section.

02 · the bet

Memory first, speed follows

tsgo (the native TypeScript 7 compiler, written in Go) settled the speed question: a native batch checker runs an order of magnitude faster than tsc on Node. What it didn't change is the memory model. Go's collector needs heap headroom — typically around 2× live data — and its AST and type objects are real heap objects: per-object headers, 64-bit pointers between them, allocation churn for the collector to chase. On the packages in the benchmark, tsgo peaks between 44 and 275 MB; and its footprint grows steeply with its parallelism (drizzle-orm: 166 → 403 MB from 1 to 8 checkers).

ztsc's answer is that a checker's live data can be tiny if nothing about the design fights it: 16 bytes per AST node, 5 per token, 28 per type, ~190 bytes of heap per source line all-in. There is no collector, so footprint equals live data — 8–21% of tsgo's peak on real packages, nearly flat as parallelism grows. Memory is the headline goal (the standing gate is ≤ 50% of tsgo; measured reality is far under it) with speed parity the constraint — and in practice ztsc is faster too, because the same choices that shrink memory (flat arrays, no pointer-chasing, no allocator locks) are the ones that make a CPU cache happy.

Why does the footprint matter at all, when a laptop has 32 GB? Because a type checker is rarely alone: it runs per-package in monorepo CI fan-outs, in editor processes that sit resident all day, in containers with hard limits, on every save in a watch loop. A checker that does the same job in a tenth of the memory is the difference between "fits" and "swaps".

03 · one idea, five ways

Data-oriented design, applied with total consistency

Nothing in ztsc is exotic. The entire performance story is five ordinary decisions applied everywhere — the AST, the binder, the type store, the module graph — with no exceptions:

  • Indices, not pointers. Every structure is flat arrays addressed by u32. Half the size of a 64-bit pointer, and traversals stream through cache lines instead of chasing the heap.
  • Struct-of-arrays. A pass that only needs node tags touches a 1-byte-per-node array and nothing else.
  • Arenas, no frees, no GC. Allocation is a pointer bump; deallocation is free; footprint equals live data.
  • Intern everything, compare integers. Strings become atoms once; types are hash-consed into TypeIds once. Equality and cache keys are integer compares forever after.
  • Seal, then share. Each phase's output is frozen before the next phase reads it — so all parallelism runs without a single lock.

These reinforce each other: interning is only cheap because types are immutable; immutability is natural because arenas never free; lock-free parallelism is safe because everything is sealed; small nodes are possible because indices replaced pointers. Laziness is the quiet sixth decision — named types expand only when touched, per-symbol state pages in only when faulted — which is how embedding the real 14k-line TypeScript standard library costs almost nothing on a project that touches a fraction of it. The internals tour walks through each structure.

Zig is the right tool for exactly this program: manual memory management is the point, not a burden (there are three allocation patterns in the whole codebase — arena, mmap, interner); @embedFile bakes the lib into the binary; and the standard library's MultiArrayList is struct-of-arrays as a first-class citizen. The result compiles to a single static binary with zero dependencies — build.zig.zon lists none.

04 · the subset

A measured subset, not an aspirational one

A TypeScript checker that mishandles code is worse than one that refuses it. ztsc's contract from day one: anything out of subset parses into an explicit unsupported node with a clear diagnostic — never a wrong answer, never a crash (every phase is fuzzed) — and in the handful of known edge cases where ztsc can't yet reproduce an error, it under-reports rather than inventing errors on valid code.

What went into the subset was decided by measurement, not by spec order. Every unsupported node records which construct it covers, so --census over a vendored corpus of real published .d.ts (2,945 files, ~131k lines — @types/node, @types/react, @types/lodash, rxjs, drizzle-orm, zod, hono, …) is a frequency table of what real code actually needs. The first census said conditional types and infer were half of everything missing — so they were built first. Today every type-level bucket in that census reads zero, and CommonJS interop (export =, import = require) has landed as well; the residual is const-symbol computed keys, consciously deferred. Correctness of what is checked is enforced differentially: a 400-case conformance suite validated against the pinned native TypeScript compiler (tsgo 7.0.2), down to error codes and positions.

05 · determinism

Parallel, with one answer

Parallel compilers often buy speed with nondeterminism — output order that depends on thread scheduling. ztsc treats determinism as a hard guarantee instead: diagnostics print per file in discovery order, position-sorted; each file is checked by exactly one checker instance; even rendered type display order is structural rather than dependent on internal ids. The conformance suite asserts byte-identical output for any --workers / --checkers value. This is what makes the parallelism honest: you can change the parallelism dial for your machine and diff the output.

It also explains the shape of the parallelism. N independent checker instances that re-derive some types their siblings also built is redundant work on paper — but it needs zero synchronization, keeps every structure single-writer, and the measured duplication is under a quarter of a type arena that is itself under 1 MB. Sharing state would save memory ztsc doesn't miss, at the price of locks it refuses to hold.

06 · batch first

Why no watch mode or LSP yet

ztsc ships as a batch checker first because that's the piece whose value can be proven with numbers today — and because incremental modes built on a shaky substrate calcify the shakiness. The architecture was designed not to preclude them: every phase's output is an immutable sealed artifact per file, which is precisely the granularity a watch mode invalidates and an LSP queries. Both are planned next.

07 · negative results

Optimizations are gated on measurement — including the ones that lose

Two plausible optimizations were seriously pursued and measured out, and the project keeps them on the record (EXPERIMENTS.md) rather than letting them be re-attempted from scratch:

  • Pre-expanding the standard library into a shared type store — built, verified byte-identical across 17 projects, then reverted: the whole lib interns to ~90 KB / ~2,500 types, while each checker's private store is dominated by per-checker instantiations that can't be pre-shared. Lazy expansion had already made the problem not exist.
  • A pre-parsed, embedded lib blob to skip the cold-start lib parse — never built: the measurement showed the entire realizable saving is 2–4 ms, invisible below the 10 ms process-wall resolution.

The method is the point: speculative performance work is gated on a measurement first, and implementations that lose are preserved in git history via commit-then-revert, with the numbers written down — so a future change in the economics is easy to recognize.
