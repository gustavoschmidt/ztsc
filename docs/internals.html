How ZTSC works

A guided tour of the Zig TypeScript Checker — a type checker (no JS emit) for a strict-mode TypeScript subset. The whole design serves one bet: memory is where a systems language beats a garbage-collected one, and speed follows from the same choices.

01 · the pipeline

One pass, three parallel regimes

ZTSC is a batch pipeline. Files are discovered from the entry points by a single-owner completion-queue scheduler: each file's front end (load, scan, parse, bind) runs as one task on the worker pool, and the import specifiers and triple-slash /// <reference> directives it surfaces queue new file tasks immediately — no barrier between "waves", so one huge file can't stall the rest of the graph. Module resolution is memoized per (importer dir, specifier) — including a negative cache — so a bare specifier imported from a hundred files walks node_modules once. Once discovery drains, a serial link step builds sealed import/export tables, merges cross-file global declarations, and registers ambient modules; then the program is partitioned by estimated cost (greedy by AST size) across N independent checker instances that read everything lock-free.

The safety story is immutability boundaries, not locks: parse output is frozen before bind reads it; bind output is frozen before check reads it. The only shared mutable structure in the whole program is the string interner, and it is sharded.

02 · one struct per stage

The pipeline, as data

Every stage of the pipeline is a pure function from one sealed struct to the next. Each struct is plain data — slices of integers, no pointers, no methods that mutate — produced by exactly one phase, frozen at its seal boundary, and read immutably by everything downstream. The handoff chain is the architecture: what a phase is allowed to know is exactly what its input struct holds.

To make the handoffs concrete, here is one file traced through every struct. The program is a deliberate one-liner with a type error:

// demo.ts
const n: number = "hi";

load → Source (source.zig)

pub const Source = struct {
    path: []const u8,         // "demo.ts"
    bytes: []const u8,        // the 24 raw bytes, mmapped read-only — never copied
    line_starts: []const u32, // [0] — one line; line/col computed lazily from this
    backing: Backing,         // .mapped (mmap) | .owned (arena fallback) | .borrowed
};

The file's bytes stay OS-owned pages for the whole run. Everything downstream refers to them by u32 byte offset; line_starts turns an offset back into line:column only when a diagnostic is actually printed.

scan → Tokens (scanner.zig)

pub const Tokens = struct {
    tags:   []const Tag, // 1 B/token
    starts: []const u32, // 4 B/token · bit 31 = "newline before" (drives ASI)
};

For demo.ts the two arrays hold eight entries — 5 bytes per token, and no end offsets stored (they're recomputed by rescanning the token when needed):

index    0              1           2      3           4    5               6          7
tags     keyword_const  identifier  colon  identifier  eq   string_literal  semicolon  eof
starts   0              6           7      9           16   18              22         23

parse → Ast (ast.zig)

pub const NodeItem = struct {
    tag: Tag,               // 1 B — 160+ node kinds
    main_token: TokenIndex, // 4 B — index into Tokens (position comes from here)
    data: Data,             // { lhs: u32, rhs: u32 } — meaning depends on tag
};

pub const Ast = struct {
    tokens: scanner.Tokens,
    nodes: std.MultiArrayList(NodeItem).Slice, // SoA: tag[], main_token[], data[]
    extra_data: []const u32,                   // variable-length children, by range
    diagnostics: []const Diagnostic,           // parse errors (code + span)
};

The demo file becomes five nodes (indices illustrative — exact order follows parse order; node 0 is always root). Note how each tag's doc comment in ast.zig states its lhs/rhs layout — declarator_full keeps the name in lhs and points rhs into extra_data for its {type, init} pair, because two fields can't hold three children:

node  tag              main_token  data                     meaning
0     root             0           extra range → [4]        one top-level statement
1     identifier       3           —                        the type annotation `number`
2     string_literal   5           —                        the initializer "hi"
3     identifier       1           —                        the name `n`
4     var_decl_one     0 (const)   lhs = 5                  one-declarator declaration
5     declarator_full  1           lhs = 3, rhs = extra →   `n: number = "hi"`

bind → Bind (binder.zig)

pub const Bind = struct {
    // symbols — SoA, index 0 reserved
    symbol_names:  []const Atom,        // interned name per symbol
    symbol_flags:  []const SymbolFlags, // packed u32 bitset (const_decl, class, …)
    symbol_scopes: []const ScopeId,
    symbol_decls_start: []const u32,    // symbol i's decl nodes =
    symbol_decls:  []const Node,        //   decls[start[i]..start[i+1]]

    // scopes — SoA, 0 = file scope; members sorted by atom per segment
    scope_parents: []const ScopeId,
    scope_kinds:   []const ScopeKind,
    scope_members_start: []const u32,
    member_atoms:  []const Atom,        // lookup = binary search, 8 B/member
    member_syms:   []const SymbolId,

    // flow graph — SoA, 0 = none, 1 = shared unreachable
    flow_tags: []const FlowTag,         // start / assign / cond_true / branch_label…
    flow_a: []const u32,                // antecedent (or extra range start)
    flow_b: []const u32,                // AST node (or extra range end)
    flow_map_nodes: []const Node,       // identifier read → flow id, sorted;
    flow_map_ids:   []const FlowId,     //   8 B per *reference*, not per node

    imports: []const ImportRec,         // for the linker
    exports: []const ExportRec,
    // this file's global contributions (empty for a typical app module —
    // the linker then skips the file entirely)
    global_atoms: []const Atom,
    global_syms:  []const SymbolId,
    ambient_modules: []const AmbientModule, // declare module "spec" { … } blocks
};

For the demo: one symbol (names[1] = atom("n"), flags[1] = .{ .const_decl = true }, scope 0, decls → node 5), one member entry in the file scope's segment, and a two-node flow graph — start, then assign whose antecedent is start and whose b is the declarator node. When the checker later asks "what is n here?", it walks this flow chain backwards to the assignment.

resolve + link → Program (modules.zig)

pub const ProgFile = struct {      // one file's sealed front-end artifacts
    path: []const u8,
    src:  []const u8,
    tree: *const Ast,
    bind: *const Bind,
    specs: SpecMap,                // import specifier atom → FileId
};

pub const Program = struct {
    files: []const ProgFile,
    sym_base: []const u32,         // prefix sums: global sym id = sym_base[file] + local id
    links: []const FileLinks,      // per file: sorted import/export → Target tables
    globals: Globals,              // lib + script global names, atom-sorted
    merged: []const MergedSym,     // cross-file declaration merges
    ambient_exports: []const AmbientExport, // declare module "spec" export tables
};

This is the one struct the check phase receives. For single-file demo.ts it is nearly empty scaffolding (files.len = 2 counting the embedded lib, sym_base = [0, lib_syms, lib_syms+2]); in a real project the links tables answer "this import binding resolves to that file's export" in one binary search, and merged holds names like NodeJS that two-plus files reopened — a symbol-table overlay referencing the sealed per-file Binds by (file, sym), never copying them.

check → Store + Diag (types.zig, checker.zig)

pub const Store = struct {         // per checker instance — private, no locks
    kinds:  std.ArrayList(Kind),   // u8: union_type / object / function / ref / …
    data_a: std.ArrayList(u32),    // fixed payload, meaning depends on kind
    data_b: std.ArrayList(u32),
    extra:  std.ArrayList(u32),    // members / props / params, by range
    map: std.HashMapUnmanaged(TypeId, void, MapCtx, 80), // hash-consing table
};

Checking the declarator asks two questions of the store. The annotation number is a fixed intrinsic — TypeId = 8, no work at all. The initializer interns string_literal with a = atom("hi"), b = fresh — the first non-fixed type this checker makes (first_free_index). Then isAssignable(literal, number) says no, memoized on the (TypeId, TypeId) pair, and the checker emits its diagnostic struct — note it differs from the front end's 2-byte-coded diagnostics.Diagnostic by carrying a tsc code and a rendered message:

pub const Diag = struct {   // checker.zig
    code: u16,              // 2322
    file: FileId,           // demo.ts
    span: Span,             // { start: 18, end: 22 } — the "hi" bytes
    msg: []const u8,        // Type 'string' is not assignable to type 'number'.
};

report → render.zig

renderPretty joins the two ends of the chain: the Diag's span plus the original Source's line_starts produce the tsc-style excerpt — file, line:col, code frame, caret under bytes 18–22. Nothing in between ever stored a line number.

demo.ts:1:19 - error TS2322: Type 'string' is not assignable to type 'number'.

1 const n: number = "hi";

Two things to notice across all six structs. Every cross-stage reference is a u32 index into a sealed array — token index, node index, atom, symbol id, flow id, file id, type id — so "follow an edge" is always an array subscript, never a pointer chase. And each struct stores only what its stage learned: positions live in Tokens, structure in Ast, names in Bind, wiring in Program, meaning in Store — nothing is duplicated forward, which is why the whole 93k-line corpus fits in 17.8 MB of heap.

04 · the ast

No pointers, anywhere

The AST is modeled on the Zig compiler's own — a proven data-oriented design. A node is not an object; it is a u32 index. Storage is struct-of-arrays (std.MultiArrayList): three parallel arrays hold each node's tag (1 byte), main token (4 bytes) and a fixed {lhs, rhs} payload (8 bytes). Anything variable-length — statement lists, call arguments, union members — lives in one shared extra_data: []u32 side array that nodes point into by range.

Three details worth knowing before reading ast.zig:

  • The zero sentinel. Node 0 is always the root and the root is never anyone's child, so 0 means "none" for optional nodes, extra-data indices and token references alike. No maxInt sentinels, no option types bloating the arrays.
  • Strings are interned once, globally. Identifier and string-literal text becomes an Atom (u32); every later phase compares atoms, never bytes.
  • Out-of-subset syntax parses into an unsupported node spanning its token range, tagged with which construct it is (the census feeds on this), paired with a clear diagnostic — never a wrong answer, never a crash (every phase is fuzzed).

05 · the binder

Symbols, scopes and flow — same discipline

The binder produces three things per file, all in the AST's flat-array style: a symbol table, a scope tree, and a control-flow graph that the checker later walks for narrowing. It costs 33.0 bytes per source line, and its output is sealed immutable the moment binding finishes.

  • One hash map per file, not per scope. While binding, inserts and duplicate detection go through a single open-addressed map keyed by (scope << 32) | atom. At seal time members are flattened into two parallel arrays segmented per scope and sorted by atom, so lookup is a binary search costing 8 bytes/member — no per-scope hash-table headers or load-factor waste.
  • tsc-style symbols. One symbol per (scope, name) across value and type space — var x; interface x {} is one symbol with both flags. Conflicts are caught with per-kind "excludes" masks; overloads and interface merging hang off a per-symbol declaration list.
  • The flow graph is antecedent-linked, modeled on tsc: assign, cond_true/cond_false, branch and loop labels. Single-antecedent joins collapse and never allocate. Flow ids attach to identifier reads through a compact sorted (node, flow) map — 8 bytes per reference rather than 4 bytes for every node.
  • Global contributions are pay-per-use. Each Bind carries a slice of the names this file contributes to the global scope — the whole top level for a script or the lib, only declare global blocks for a module — plus its declare module "spec" ambient blocks. For a typical app module both slices are empty and the linker's cross-file merge skips the file entirely, so @types/node-style declaration merging costs nothing on code that doesn't use it. This layer passed its real-world acceptance test: checking against the actual published @types/node, ztsc's cross-file merge diagnostics match the pinned tsgo 7.0.2 oracle.

06 · types

A type is a u32, too

Types live in a struct-of-arrays store shaped exactly like the AST: a TypeId is a u32 index, fixed payloads sit in parallel arrays, variable payloads (union members, object properties, function params) go in a shared extra array. The crucial move is that types are hash-consed: structurally identical types get the same id. The store is split in two — a shared frozen base (ids below base_len, readable lock-free by every checker) and a per-checker overlay that owns everything else — substrate that made the lib-sharing experiments in the negative-results note cheap to run.

Everything downstream gets cheaper because of this. Well-known types sit at fixed indices (any = 1 … empty_object = 16), so no lookup is needed for the common cases. Unions are canonical at construction — flattened, deduped, never dropped, literals absorbed by their base primitive, true | false collapsed to boolean — which means the checker compares canonical forms, not spellings. Display order is structural rather than TypeId-based, so a rendered union reads the same for any checker count and any id assignment. Named types (interfaces, classes, aliases) stay lazy as .ref {symbol, args} until the checker actually needs their members — which is also why embedding the full 14k-line lib barely moved RSS: only the touched surface ever interns. Object-literal "freshness" for excess-property checks is one flag bit that participates in interning, so no side table is needed.

The checker itself (checker.zig, the largest file at 11k lines) implements structural assignability, inference, declared-overload resolution, control-flow narrowing (truthiness, typeof, equality, in, instanceof, discriminants, assignments), user-defined type guards and assertion functions, enums, accessors, as const/satisfies, this-typing, TC39 decorators with signature checks, unique symbol, import() types, async/await/Promise typing, the Symbol.iterator protocol, JSX element typing — and the type-level core: conditional types with infer and distributivity, mapped types with as key remapping, template-literal types and the string intrinsics, indexed access and keyof over generics, recursive aliases, and call/construct signatures in interfaces and type literals. Generic instantiation is memoized with depth limits so that machinery can't blow up. All of it matches the TypeScript compiler's observable behavior, verified differentially.

07 · memory

Allocation lifetime = phase lifetime

There is no general-purpose heap churn in ZTSC. The rule, followed everywhere: one arena per lifetime, and no per-object frees, ever. Every structure is bump-allocated into the arena matching its phase, sealed read-only when the phase ends, and released wholesale when the process exits. Since arenas are per-thread, there is no allocator contention either — no locks in the malloc path, because there is no malloc path.

Why this wins against tsgo specifically: Go's collector needs heap headroom (typically ~2× live data), and its AST/type objects carry per-object headers and pointers. ZTSC's live data is its footprint — the entire heap (all arenas plus the interner) for the 93k-line corpus is 17.8 MB, about 190 bytes per source line, all-in. That figure used to be 45.5 MB: a dedicated debt-paydown pass (seal the parser's output into exact-size arrays instead of stranding doubling-realloc slack in the arena, drop a duplicate token scan, reset the resolution scratch per file) cut arena heap by more than half without touching any per-structure cost. Source bytes never even count: files are mmapped read-only and the AST references them by offset; the only copies made are into the string interner.

The same discipline covers the one place a checker's state must span the whole program: its per-symbol arrays (sym_types/sym_state) are ZeroPagedArrays (zeropage.zig) — fixed-length views over fresh MAP_ANONYMOUS pages, the one source where kernel zero-fill is a documented guarantee. The all-zero "not computed" state costs no initialization, and untouched pages never become resident, so a checker's residency tracks its working set, not the program's symbol count. That's why peak RSS no longer grows monotonically with --checkers.

08 · parallelism

N checkers, zero locks, one answer

The front end is embarrassingly parallel: one task per file on a std.Thread.Pool, each task owning its own arenas. The interesting part is the check phase, which borrows tsgo's key trick: partition the program's files across N independent checker instances (default 4) — greedily by estimated cost (AST node count), so one giant file doesn't leave three checkers idle (on the skewed stress corpus the cost-based partition cut check time 26%; on bigfan, 2.4×). Each checker owns a private type store and private caches, and reads the sealed AST/binder/link data freely. The cost is that checkers re-derive some types their siblings also built; the payoff is parallelism with no synchronization at all.

That duplication is the project's main speed/memory dial, and it is measured rather than guessed:

Determinism is a hard guarantee, not a hope: every diagnostic is tagged with its file, each file is checked by exactly one checker, output is printed per file in discovery order, position-sorted, and even rendered type display order is structural rather than id-dependent. The conformance suite asserts byte-identical output for any N.

09 · synthesis

Why it's fast (and small)

Nothing in ZTSC is exotic. The performance comes from five decisions applied with total consistency — the same discipline in the AST, the binder, the types and the module graph:

  • Indices, not pointers. Every structure is flat arrays addressed by u32. Half the memory of a 64-bit pointer, and traversals stream through cache lines instead of chasing the heap.
  • Struct-of-arrays. A pass that only needs node tags touches a 1-byte-per-node array and nothing else. 16 B/node vs 100–200+ for object ASTs.
  • Arenas, no frees, no GC. Allocation is a pointer bump; deallocation is free; footprint equals live data — no collector headroom, no per-object headers.
  • Intern everything, compare integers. Strings become atoms once; types become TypeIds once. Equality checks and cache keys are integer compares for the rest of the run.
  • Seal, then share. Each phase's output is frozen before the next phase reads it, so all parallelism — worker pool and N checkers — runs without a single lock.

Notice these reinforce each other: interning only works cheaply because types are immutable; immutability is natural because arenas never free; lock-free parallelism is safe because everything is sealed; and small nodes are possible because indices replaced pointers. It's one idea — data-oriented design — expressed five ways. Laziness is the sixth, quieter decision: named types expand only when touched, per-symbol state pages in only when faulted — which is how a 12k-line embedded lib costs almost nothing on a project that uses a fraction of it.

The first census counted 2,287 unsupported constructs, half of them conditional types and infer; today every type-level bucket reads zero, and CommonJS interop (export =, import = require) has landed as well, and const-symbol computed keys (drizzle's [entityKind], typebox's [Kind]) now parse and check — the census residual is a dozen member-expression computed keys ([ns.member]). The acceptance bar was the real, published @types/node: ztsc's cross-file merge diagnostics match the TypeScript compiler on it (pinned tsgo 7.0.2 oracle).

What remains out for v0.0.1 is a short, deliberate list: a couple of lenient CommonJS corners that degrade to any instead of erroring, merged-interface iterators (for…of over a DOM collection whose [Symbol.iterator] is added by a second interface declaration, e.g. URLSearchParams/Headers, and for await over AsyncIterable; single-declaration iterables like NodeListOf work), member-expression computed keys ([ns.member] such as [EventEmitter.captureRejectionSymbol]; a const-unique symbol key written as a bare identifier — [entityKind], [Kind] — is checked with full cross-file nominal identity), JSX leniencies (the JSX namespace resolves from the real @types/react and spread attributes are checked, but prop-type mismatches arriving inside a spread object, spreads of unions/generics, and children value typing under-report rather than erroring), and watch mode / LSP — both planned next, on the sealed-phase architecture that was designed not to preclude them. Unsupported syntax still produces a clear "not supported" diagnostic — never a wrong answer, never a crash — and in the handful of known miss cases ztsc under-reports rather than inventing errors on valid code.

Everything before release is done: the semantic breadth, the instantiation discipline, the type-level core, a correctness-debt paydown, the real lib and the real @types/node acceptance, the substrate experiments, and pre-publish polish — MIT license, a NOTICE crediting Microsoft's TypeScript (Apache-2.0) for the embedded lib and diagnostic strings, and the README/benchmark rewrite. What's left is the ship itself: v0.0.1 as a working bunx ztsc.

Negative results, preserved. Two optimizations were seriously pursued and measured out — recorded in EXPERIMENTS.md so the economics are easy to re-check later. The frozen-base lib payload (pre-expand the whole lib once into the shared base store) was built, verified byte-identical against the old binary across 17 projects, and reverted: the entire lib is only ~90 KB / ~2,500 types, while each checker's overlay runs 0.7–1.4 MB dominated by per-checker instantiations that can't be pre-expanded — and lazy expansion already held RSS at ~26% of tsgo, so there was no shortfall to claw back. The pre-parsed lib blob was never built: the entire cold-start lib cost is ≈8.4 ms, of which a blob eliminates 2–4 ms — invisible below the 10 ms process-wall resolution. Method both times: gate speculative perf work on a measurement first, and preserve built implementations via commit-then-revert.
