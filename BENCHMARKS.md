# ZTSC v0.0.1 — Benchmark Report

The release story for v0.0.1 (PLAN.md M6): how a data-oriented,
arena-allocated, parallel checker in Zig compares against `tsc` 5.5.4 and
`tsgo` (the TypeScript 7 native preview) — **on the ztsc v0.0.1 subset**
(see the caveats in §6 before quoting any of these numbers).

Headline, multi corpus (201 files / 93,364 lines, cross-importing module
graph):

| | wall (median) | peak RSS |
|---|---:|---:|
| **ztsc** (`--checkers=4`) | **0.02 s** | **71.9 MB** |
| tsgo 7.0.0-dev.20260707.2 | 0.08 s | 204.5 MB |
| tsc 5.5.4 | 0.91 s | 314.4 MB |

Both PLAN acceptance gates pass (§5): wall well within 1.25× of tsgo
(~0.25×), peak RSS ≈ 35% of tsgo (target ≤ 50%).

---

## 1. Methodology

**Hardware / OS.** Apple M4 (10 cores: 4P + 6E), 32 GB RAM, macOS 26.5.1.

**Versions.**

| tool | version | how it runs |
|---|---|---|
| ztsc | 0.0.1, `zig build bench` (ReleaseFast), Zig 0.16.0 | native binary |
| tsc | 5.5.4 | Node 22.12.0 (`node .../bin/tsc`) |
| tsgo | 7.0.0-dev.20260707.2 | native arm64 binary (`@typescript/native-preview-darwin-arm64/lib/tsgo`, invoked directly — no Node wrapper in the measurement) |

**Corpora.** Deterministic synthetic projects generated by
`bench/gen_corpus.js` (seeded LCG; regenerate with `node bench/gen_corpus.js`).
All code stays inside the ztsc v0.0.1 subset (PLAN.md §5) and is lib-free
(no globals / primitive methods, since ztsc has no `lib.d.ts` yet):

| corpus | files | lines | shape |
|---|---:|---:|---|
| small | 10 | 5,520 | flat modules |
| medium | 50 | 49,600 | flat modules, `include`-driven tsconfig |
| multi | 201 | 93,364 | 5-layer cross-import graph, discovered from a single `entry.ts` via `files` |

All three tools check the same project through its `tsconfig.json`
(`-p bench/corpus/<name>`, `--noEmit` semantics) and all three report
**zero diagnostics** on every corpus (cross-checked before measuring).

**Measurement.** `bench/e2e.sh <corpus>`: one untimed warm-up run (also
warms the FS cache), then 5 timed runs under `/usr/bin/time -l`; the table
reports the **median wall clock** and **max peak RSS** of the 5 runs.
`/usr/bin/time` wall resolution is 10 ms, so ztsc's small-corpus times
saturate the clock (its internal `--timing` totals are given alongside).
Per-phase and memory numbers come from a single `--timing --memory` run of
the ReleaseFast binary (they are stable run-to-run to ~±5%).

```sh
RUNS=5 TSC=/path/to/typescript/bin/tsc TSGO=/path/to/native/tsgo \
  bench/e2e.sh medium && bench/e2e.sh multi
zig-out/bench/ztsc --pretty=false --timing --memory -p bench/corpus/multi
```

## 3. ztsc detail

### 3.1 Per-phase timing (multi corpus, `--checkers=4`, one `--timing` run)

| phase | ms | lines/s | MB/s |
|---|---:|---:|---:|
| load | 2.79 | 33.5 M | 802 |
| scan | 1.85 | 50.4 M | 1206 |
| parse | 3.49 | 26.8 M | 641 |
| bind | 5.75 | 16.2 M | 388 |
| resolve | 1.16 | — | — |
| link | 1.87 | — | — |
| check | 15.94 | 5.9 M | 140 |
| **total** | **32.9** | | |

Medium corpus, same shape: load 1.19 / scan 0.56 / parse 1.14 / bind 1.80 /
resolve 0.01 / link 0.71 / check 7.25, total 12.7 ms.

Check dominates (~50% of wall); the four checker instances finish within
1% of each other (15.74–15.90 ms), so the round-robin partition balances
well on this corpus.

### 3.2 Checker scaling and the duplicated-type dial (`--checkers=N`)

PLAN §2.3: N independent checker instances trade duplicated type
construction for lock-free parallelism. Both axes, measured:

**multi (93k lines)**

| N | wall | check ms | types created | type-arena bytes | dup overhead vs N=1 | peak RSS |
|---:|---:|---:|---:|---:|---:|---:|
| 1 | 0.05 s | 42.1 | 21,333 | 604,206 | — | 67.2 MB |
| 2 | 0.03 s | 22.9 | 22,861 | 648,003 | +7.2% | 69.4 MB |
| 4 | 0.02 s | 12.6 | 24,642 | 696,686 | +15.5% | 71.9 MB |
| 8 | 0.02 s | 10.2 | 26,283 | 737,963 | +23.2% | 69.2 MB |

**medium (50k lines)**

| N | wall | check ms | types created | type-arena bytes | dup overhead vs N=1 | peak RSS |
|---:|---:|---:|---:|---:|---:|---:|
| 1 | 0.02 s | 22.4 | 12,070 | 338,659 | — | 36.3 MB |
| 2 | 0.01 s | 11.8 | 12,401 | 344,555 | +2.7% | 36.9 MB |
| 4 | 0.01 s | 6.9 | 12,831 | 352,363 | +6.3% | 38.1 MB |
| 8 | 0.01 s | 5.6 | 13,332 | 361,868 | +10.5% | 37.7 MB |

Check-phase speedup is near-linear to N=4 (3.3×) and tapering at N=8
(4.1× on multi); the memory cost of duplication stays under a quarter of
the type arena even at N=8, and the type arena itself is <1 MB — noise
against the ~70 MB process RSS. N=4 (the default) is the sweet spot.

## 4. Cross-checking correctness while benchmarking

Before timing, all three tools were run on all three corpora:
`ztsc -p`, `tsc --noEmit -p`, and `tsgo --noEmit -p` all exit 0 with zero
diagnostics. The 180-case conformance suite (code + line differential vs
tsc 5.5.4, including 30 multi-file cases) is green at this commit.

## 5. PLAN acceptance gate (honest evaluation)

PLAN M6 gate: *on medium + large corpora, wall clock within 1.25× of tsgo
and peak RSS ≤ 50% of tsgo; conformance suite green.*

| corpus | wall vs tsgo (≤ 1.25×) | RSS vs tsgo (≤ 50%) | verdict |
|---|---|---|---|
| medium | 0.01 s vs 0.04 s ≈ **0.25×** | 38.1 / 108.9 MB ≈ **35%** | **pass** |
| multi | 0.02 s vs 0.08 s ≈ **0.25×** | 71.9 / 204.5 MB ≈ **35%** | **pass** |

Conformance: 180/180 vs tsc 5.5.4 — green.

Honesty notes:

- The PLAN's "large" corpus (~500k LOC filtered real-world snapshot) was
  never built — real-world code that stays inside the v0.0.1 subset is
  rare (PLAN §6 predicted this). The gate is evaluated on **medium +
  multi** (up to 93k LOC / 201 files). The large-corpus evaluation carries
  to the next release.
- Wall-clock ratios at these speeds sit near `/usr/bin/time`'s 10 ms
  resolution; the honest claim is "ztsc completes in a fraction of tsgo's
  wall time on these corpora", not a precise 4×.
- These corpora are synthetic and subset-restricted — see §6.

## 6. Caveats — what these numbers do and don't say

This is a **fair-for-the-subset** comparison, not a general claim that
ztsc is faster than tsc or tsgo:

- **ztsc checks a subset** (PLAN.md §5). No conditional/mapped/
  template-literal types, no `infer`, no declaration merging, namespaces,
  enums, decorators, or JSX. On real-world code, tsc and tsgo do
  substantially more semantic work per line than anything in these corpora.
- **No `lib.d.ts`.** ztsc has no standard-library types yet, and the
  corpora are deliberately lib-free. tsc/tsgo still *load and bind their
  default libs* in these runs (tsgo's `"lib": ["esnext"]` parse is real
  work ztsc skips) — some of their RSS/wall gap is that fixed cost. That
  fixed cost shrinks in relative terms as corpora grow; the multi-corpus
  gap (72 vs 204 MB) is not explained by lib loading alone.
- **The corpora are synthetic** — generated, regular code with a realistic
  import graph but less type-level diversity than human code. Relation
  cache hit rates and type dedup likely flatter ztsc slightly.
- tsc runs on Node (JIT warm-up and GC headroom are part of its numbers;
  that *is* how users run it, so it's reported as such).
- Single machine, single OS. No Linux numbers yet.

## 7. Reproducing

```sh
zig build test                      # unit + conformance (needs tsc for regen only)
node bench/gen_corpus.js            # regenerate corpora (deterministic)
RUNS=5 TSC=path/to/typescript/bin/tsc TSGO=path/to/native/tsgo bench/e2e.sh medium
RUNS=5 TSC=path/to/typescript/bin/tsc TSGO=path/to/native/tsgo bench/e2e.sh multi
zig-out/bench/ztsc --timing --memory --pretty=false -p bench/corpus/multi
```

`bench/e2e.sh` autodetects `tsc`/`tsgo` on PATH; point `TSGO` at the
native platform binary (`@typescript/native-preview-<platform>/lib/tsgo`)
rather than the npm `bin/tsgo` Node shim so RSS is measured on the real
process.
